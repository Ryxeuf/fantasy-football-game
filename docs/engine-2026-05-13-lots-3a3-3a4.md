# Lots 3.A.3 + 3.A.4 — events enrichis + narrator roster-aware (2026-05-13)

> Après les 3 fix du full driver
> ([engine-fix-2026-05-13-full-driver.md](./engine-fix-2026-05-13-full-driver.md)),
> le mode "vraie sim IA-vs-IA avec actions individuelles" tourne mais
> émet des events squelettiques (BLOCK / PASS / DODGE / TURNOVER) avec
> des `playerId` génériques (`A1`, `B1`).
>
> Ces deux lots enrichissent la timeline (granularité narrative) et
> permettent au narrator d'utiliser les vrais noms du roster.

## Lot 3.A.3 — events enrichis

Trois nouveaux `EventType` ajoutés au catalogue partagé
[match-event.ts](../packages/shared-types/src/match-event.ts) :

| Event | Émis quand | Meta |
|---|---|---|
| `PLAYER_ACTIVATION` | `state.selectedPlayerId` passe à un nouveau id non-null | `playerId`, `team` |
| `BLITZ_DECLARED` | `move.type === 'BLITZ'` (avant le BLOCK qui suit) | `attackerId`, `defenderId` |
| `KNOCKDOWN` | Joueur `active` → `stunned` (defender down sans armor break) | `playerId`, `team`, `causedBy?` |

Ordre d'émission dans `diffStatesToEvents` :
1. `PLAYER_ACTIVATION` (avant l'action du joueur)
2. `BLITZ_DECLARED` (si move=BLITZ)
3. `BLOCK` / `PASS` / `DODGE` / `FOUL` (event Move-specific existant)
4. `TD` / `KO` / `CASUALTY` / `KNOCKDOWN` (side-effects state diff)
5. `TURNOVER` / `HALFTIME` / `END` / `TURN_START` (transitions de phase)

Pourquoi cet ordre : les events partagent le même `displayAtMs`, donc
le narrator les regroupe naturellement. L'ordre logique
"activation → déclaration → résolution → conséquences" donne une
timeline lisible.

`KNOCKDOWN` n'est PAS émis si le joueur passe direct en `knocked_out`
ou `casualty` (les events `KO` / `CASUALTY` portent déjà la sémantique
"tombé + sorti du jeu"). Le KNOCKDOWN couvre uniquement le cas
intermédiaire stun-on-field.

Histogramme observé (pit-smashers vs dal-vipers, 6 seeds) :

```
PLAYER_ACTIVATION   11-28 / match
BLITZ_DECLARED      3-6   / match
KNOCKDOWN           0-1   / match
```

## Lot 3.A.4 — narrator roster-aware

[narrator.ts](../packages/sim-engine/src/replay/narrator.ts) accepte
maintenant `NarrateOptions.rosters: { home, away }` (chaque équipe =
`readonly SimRosterPlayer[]`). Quand fourni :

- Helper `formatPlayer(id, rosterIndex)` résout `playerId` → `"Bob
  Smasher (#3 Blitzer)"`.
- Mode legacy (sans rosters) : fallback sur l'id brut (`"A1"`),
  comportement inchangé pour les replays archetype-vs-archetype.

Exemples avant / après :

```
# AVANT (legacy archetype)
  BLOCK — A1 blocks B1: dice [], chose ?, ?.

# APRÈS (roster-aware)
  BLOCK — Bob Smasher (#3 Blitzer) blocks Carla Catcher (#7 Catcher):
         dice [], chose ?, ?.

  BLITZ! — Bob Smasher (#3 Blitzer) charges Carla Catcher (#7 Catcher).
  KNOCKDOWN — Carla Catcher (#7 Catcher) is knocked down by Bob
              Smasher (#3 Blitzer) (stunned).
  > Bob Smasher (#3 Blitzer) (home) takes action.
```

## Compat

- Les replays produits avant ces lots restent **décodables** : le
  catalogue n'a fait qu'ajouter des types, les anciens types
  fonctionnent à l'identique.
- Les consumers (frontend ticker, broadcaster) qui ne connaissent pas
  les nouveaux types les voient avec un fallback `default: ev.type` —
  pas de crash, juste un rendu textuel minimal pour les anciens
  clients.
- Le `narrateMatch` standalone sans `rosters` rend exactement comme
  avant — utilisé par le CLI [`pnpm sim:replay`](../packages/sim-engine/scripts/replay.ts).
